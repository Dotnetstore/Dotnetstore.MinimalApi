Minimal API sample built with .NET 10, backed by automated GitHub Actions workflows for continuous integration, artifact-based delivery, semantic versioning, changelog generation, draft release notes, coverage publishing, contribution policy checks, dependency automation, GHCR container publishing, and optional Azure App Service deployment.

## Solution structure

- `src/Dotnetstore.MinimalApi.Api.WebApi` - the Minimal API application
- `tests/Dotnetstore.MinimalApi.Api.WebApi.Tests` - automated tests for startup, middleware, CORS, and application bootstrapping
- `Helpers/TestHttp.cs` in the test project - shared HTTP test helpers
- `.github/workflows` - GitHub Actions CI/CD workflows
- `.github/ISSUE_TEMPLATE` and `.github/pull_request_template.md` - contribution templates
- `.github/DISCUSSION_TEMPLATE` - GitHub Discussions starter forms
- `.github/CODEOWNERS` - automatic reviewer ownership rules
- `docs/branch-protection.md` - recommended branch protection rules

## Contribution conventions

This repository now enforces a lightweight Conventional Commits workflow:

- pull request titles are validated by `.github/workflows/pr-title.yml`
- commit messages are validated by `.github/workflows/commit-message.yml`
- pull requests are path-labeled by `.github/workflows/auto-label.yml`
- stale issues and pull requests are managed by `.github/workflows/stale.yml`
- semantic releases are generated by `.github/workflows/release-please.yml`
- draft release notes are maintained by `.github/workflows/release-drafter.yml`

Examples:

- `feat: add order endpoint`
- `fix(api): correct CORS preflight handling`
- `test(webapi): add startup coverage`

GitHub also provides:

- issue templates for bugs and feature requests
- discussion templates for ideas, Q&A, and show-and-tell posts
- a pull request template with local validation reminders
- CODEOWNERS-based review requests
- branch protection guidance in `docs/branch-protection.md`

## Local development

The repository includes a `global.json` file that pins the .NET SDK used locally and in GitHub Actions.

### Restore, build, and test

```powershell
dotnet restore .\Dotnetstore.MinimalApi.slnx
dotnet build .\Dotnetstore.MinimalApi.slnx --configuration Release --no-restore
dotnet run --project .\tests\Dotnetstore.MinimalApi.Api.WebApi.Tests\Dotnetstore.MinimalApi.Api.WebApi.Tests.csproj --configuration Release --no-build
```

### Generate a local coverage report

```powershell
dotnet tool install dotnet-coverage --tool-path .\.tools
dotnet tool install dotnet-reportgenerator-globaltool --tool-path .\.tools
.\.tools\dotnet-coverage collect "dotnet run --project .\tests\Dotnetstore.MinimalApi.Api.WebApi.Tests\Dotnetstore.MinimalApi.Api.WebApi.Tests.csproj --configuration Release --no-build" -f cobertura -o .\artifacts\coverage\coverage.cobertura.xml
.\.tools\reportgenerator "-reports:.\artifacts\coverage\coverage.cobertura.xml" "-targetdir:.\artifacts\coverage-report" "-reporttypes:HtmlInline;MarkdownSummaryGithub;Badges"
```

### Run the Web API locally

```powershell
dotnet run --project .\src\Dotnetstore.MinimalApi.Api.WebApi\Dotnetstore.MinimalApi.Api.WebApi.csproj
```

### Build the container image locally

```powershell
docker build -t dotnetstore-minimalapi:local .
```

## Web API configuration

Runtime behavior for CORS, HSTS, HTTPS redirection, and rate limiting is configured under the `WebApi` section in `src/Dotnetstore.MinimalApi.Api.WebApi/appsettings.json`.

```json
{
  "WebApi": {
	"Cors": {
	  "AllowedOrigins": ["http://localhost:7000"],
	  "AllowedMethods": ["GET", "POST", "PUT"]
	},
	"Hsts": {
	  "Preload": true,
	  "IncludeSubDomains": true,
	  "MaxAgeDays": 30
	},
	"HttpsRedirection": {
	  "Development": {
		"RedirectStatusCode": 307,
		"HttpsPort": 7201
	  },
	  "Production": {
		"RedirectStatusCode": 308,
		"HttpsPort": 443
	  }
	},
	"RateLimiting": {
	  "RejectionStatusCode": 429,
	  "RejectionMessage": "Too many requests. Please try again later.",
	  "PartitionKeyFallback": "unknown",
	  "GlobalPermitLimit": 50,
	  "GlobalQueueLimit": 10,
	  "GlobalWindowSeconds": 15,
	  "ShortPermitLimit": 10,
	  "ShortQueueLimit": 0,
	  "ShortWindowSeconds": 15
	}
  }
}
```

### Validation rules

The application validates `WebApi` options during startup and fails fast when configuration is malformed. In practice this means:

- `Cors.AllowedOrigins` and `Cors.AllowedMethods` must contain at least one non-empty value
- `Hsts.MaxAgeDays` must be greater than `0`
- HTTPS redirect status codes must be redirect responses (`300`-`399`)
- HTTPS ports must be between `1` and `65535`
- rate limiter permit/window values must be positive
- queue limits cannot be negative
- `RejectionMessage` and `PartitionKeyFallback` cannot be blank

If you override these values with environment-specific settings, keep the same `WebApi` section structure.

## GitHub Actions

### Continuous Integration

The `CI` workflow runs on pushes to `main` and on pull requests targeting `main`.

It performs the following steps:

1. checks out the repository
2. installs the SDK version from `global.json`
3. restores the solution
4. builds the solution in `Release`
5. runs the automated test project with coverage collection
6. generates an HTML and Markdown coverage report
7. uploads the coverage artifacts for download from the workflow run

Workflow file: `.github/workflows/ci.yml`

The CI run publishes:

- a Cobertura XML file
- a browsable HTML coverage report
- a GitHub job summary with coverage details

### Continuous Delivery

The `CD` workflow runs on:

- pushes to `main`
- version tags that match `v*`
- manual dispatch from GitHub Actions

It performs the following steps:

1. restores, builds, and tests the solution
2. publishes the Web API in `Release`
3. uploads the published output as a GitHub Actions artifact
4. when triggered by a `v*` tag, creates or updates a GitHub release with a zipped publish artifact
5. when Azure deployment secrets are configured, deploys `main` to a staging slot and tags or manual production promotions to the production slot

Workflow file: `.github/workflows/cd.yml`

### Semantic versioning and changelog generation

The `Release Please` workflow automates semantic versioning and changelog generation from Conventional Commits.

It does the following:

1. creates or updates a release PR on `main`
2. updates `CHANGELOG.md` in the release PR
3. creates a Git tag and GitHub release when the release PR is merged

Workflow and configuration files:

- `.github/workflows/release-please.yml`
- `.release-please-config.json`
- `.release-please-manifest.json`

### Draft release notes

The `Release Drafter` workflow keeps a draft release up to date using merged pull requests and labels.

Workflow and configuration files:

- `.github/workflows/release-drafter.yml`
- `.github/release-drafter.yml`

### Container publishing

The `Container Publish` workflow builds and pushes a container image to GitHub Container Registry.

It runs on:

- pushes to `main`
- version tags that match `v*`
- manual dispatch from GitHub Actions

Published image:

```text
ghcr.io/dotnetstore/dotnetstore-minimalapi
```

Workflow file: `.github/workflows/container-publish.yml`

### Dependabot

Dependabot is configured in `.github/dependabot.yml` and checks for updates every week for:

- NuGet dependencies
- GitHub Actions workflow dependencies

## Azure deployment configuration

The `CD` workflow includes an optional Azure App Service deployment job.

The deployment job is automatically skipped unless these repository secrets are configured:

- `AZURE_CLIENT_ID`
- `AZURE_TENANT_ID`
- `AZURE_SUBSCRIPTION_ID`
- `AZURE_WEBAPP_NAME`

This uses GitHub OIDC with `azure/login` and deploys the published Web API artifact to Azure App Service.

Staging slot behavior:

- pushes to `main` deploy to the `staging` slot by default
- you can override the slot name with the repository variable `AZURE_STAGING_SLOT_NAME`

Production behavior:

- tags matching `v*` deploy to the production slot
- manual dispatch lets you choose `staging` or `production`

## Release convention

To create a versioned release artifact, push a Git tag such as:

```powershell
git tag v1.0.0
git push origin v1.0.0
```

That tag triggers the `CD` workflow, uploads the publish output, and creates a GitHub release archive.

The same tag also triggers the container publishing workflow so a matching container tag is pushed to GHCR.

Release Please can also create version tags and GitHub releases automatically when its release PR is merged.
